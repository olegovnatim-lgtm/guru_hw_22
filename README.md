# Тест-кейсы: Book Club API

## Принцип разделения

Каждый позитивный сценарий разбит на **два тест-кейса**:

1. **Status + Schema** — проверяет статус-код, наличие всех обязательных полей, типы данных, отсутствие лишних полей. Если этот тест падает — проблема на уровне контракта API.
2. **Content** — проверяет, что значения полей корректны (не заглушки, не stub, соответствуют отправленным данным). Если этот тест падает при зелёном Schema — данные неверны, но структура цела.

Негативные сценарии остаются **одним тест-кейсом** (проверяем статус-код + сообщение об ошибке), т.к. при ошибке нет «контента» для отдельной проверки.

## 1. Регистрация — `POST /api/v1/users/register/`

### TC-REG-01: Успешная регистрация — status + schema

**Предусловия:** username `newuser_01` не занят.

**Тело запроса:**
```json
{ "username": "newuser_01", "password": "SecurePass123!" }
```

**Проверки:**
- Status code: `200`
- Тело содержит поля: `username` (string), `password` (string)
- Нет лишних полей (например, `token`, `id`, `email`)

---

### TC-REG-02: Успешная регистрация — content

**Предусловия:** те же, что в TC-REG-01.

**Проверки:**
- `username` == `"newuser_01"`

---

### TC-REG-03: Регистрация с уже занятым username

**Предусловия:** `newuser_01` уже зарегистрирован.

**Тело запроса:**
```json
{ "username": "newuser_01", "password": "OtherPass456!" }
```

**Проверки:**
- Status code: `400`
- Ответ содержит ошибку, связанную с полем `username`

---

### TC-REG-04: Регистрация без поля `password`

**Тело запроса:**
```json
{ "username": "user_no_pass" }
```

**Проверки:**
- Status code: `400`
- Ошибка валидации для поля `password`

---

### TC-REG-05: Регистрация без поля `username`

**Тело запроса:**
```json
{ "password": "SomePass789!" }
```

**Проверки:**
- Status code: `400`
- Ошибка валидации для поля `username`

---

### TC-REG-06: Регистрация с пустым телом

**Тело запроса:**
```json
{}
```

**Проверки:**
- Status code: `400`
- Ошибки валидации для `username` и `password`

---

### TC-REG-07: Username с недопустимыми символами

**Тело запроса:**
```json
{ "username": "user name!#$", "password": "ValidPass123!" }
```

**Проверки:**
- Status code: `400`
- Ошибка валидации `username` (паттерн `^[\w.@+-]+$`)

---

### TC-REG-08: Username длиной 151 символ (выше maxLength=150)

**Тело запроса:** `username` = `"a" * 151`, `password` валидный.

**Проверки:**
- Status code: `400`
- Ошибка валидации `username` по длине

---

### TC-REG-09: Username длиной ровно 150 символов — status + schema

**Тело запроса:** `username` = `"a" * 150`, `password` валидный.

**Проверки:**
- Status code: `200`
- Тело содержит `username` (string)

---

### TC-REG-10: Username длиной ровно 150 символов — content

**Проверки:**
- `username` имеет длину 150 символов, совпадает с отправленным

## 2. Авторизация — `POST /api/v1/auth/token/`

### TC-AUTH-01: Успешная авторизация — status + schema

**Предусловия:** пользователь `testuser` зарегистрирован с паролем `SecurePass123!`.

**Тело запроса:**
```json
{ "username": "testuser", "password": "SecurePass123!" }
```

**Проверки:**
- Status code: `200`
- Тело содержит поля: `access` (string), `refresh` (string)
- Нет лишних полей (например, `user`, `id`, `email`)

---

### TC-AUTH-02: Успешная авторизация — content

**Проверки:**
- `access` — валидный JWT (3 сегмента через `.`)
- `refresh` — валидный JWT (3 сегмента через `.`)

---

### TC-AUTH-03: Неверный пароль

**Тело запроса:**
```json
{ "username": "testuser", "password": "WrongPassword!" }
```

**Проверки:**
- Status code: `401`
- Ответ НЕ содержит полей `access` / `refresh`
- Сообщение об ошибке аутентификации

---

### TC-AUTH-04: Несуществующий username

**Тело запроса:**
```json
{ "username": "ghost_user_xyz", "password": "AnyPass123!" }
```

**Проверки:**
- Status code: `401`
- Текст ошибки не раскрывает, что именно неверно (username или password)

---

### TC-AUTH-05: Отсутствует поле `password`

**Тело запроса:**
```json
{ "username": "testuser" }
```

**Проверки:**
- Status code: `400`
- Ошибка валидации для `password`

---

### TC-AUTH-06: Отсутствует поле `username`

**Тело запроса:**
```json
{ "password": "SecurePass123!" }
```

**Проверки:**
- Status code: `400`
- Ошибка валидации для `username`

---

### TC-AUTH-07: Пустое тело запроса

**Тело запроса:**
```json
{}
```

**Проверки:**
- Status code: `400`
- Ошибки для `username` и `password`

---

### TC-AUTH-08: Пустые строки в полях

**Тело запроса:**
```json
{ "username": "", "password": "" }
```

**Проверки:**
- Status code: `400`
- Ошибки валидации — поля не могут быть пустыми

---

### TC-AUTH-09: Refresh токена — status + schema (`POST /api/v1/auth/token/refresh/`)

**Предусловия:** получен валидный `refresh` через TC-AUTH-01.

**Тело запроса:**
```json
{ "refresh": "<valid_refresh_token>" }
```

**Проверки:**
- Status code: `200`
- Тело содержит поле `access` (string)
- Нет лишних полей

---

### TC-AUTH-10: Refresh токена — content

**Проверки:**
- `access` — валидный JWT

---

### TC-AUTH-11: Refresh с невалидным токеном

**Тело запроса:**
```json
{ "refresh": "invalid.token.here" }
```

**Проверки:**
- Status code: `401`
- Сообщение о невалидном/истёкшем токене

---

### TC-AUTH-12: Logout — `POST /api/v1/auth/logout/`

**Предусловия:** получен валидный `refresh`.

**Тело запроса:**
```json
{ "refresh": "<valid_refresh_token>" }
```

**Проверки:**
- Status code: `200`
- Повторный `POST /api/v1/auth/token/refresh/` с этим же токеном → `401`

## 3. CRUD Клубов

### 3.1 Создание — `POST /api/v1/clubs/`

#### TC-CLUB-C01: Успешное создание — status + schema

**Предусловия:** пользователь авторизован.

**Тело запроса:**
```json
{
  "bookTitle": "Война и мир",
  "bookAuthors": "Лев Толстой",
  "publicationYear": 1869,
  "description": "Классика русской литературы",
  "telegramChatLink": "https://t.me/war_and_peace"
}
```

**Проверки:**
- Status code: `201`
- Тело содержит обязательные поля: `id` (int), `bookTitle` (string), `bookAuthors` (string), `publicationYear` (int), `description` (string), `telegramChatLink` (string), `owner` (int), `members` (array of int), `reviews` (array), `created` (datetime string), `modified` (datetime/null)
- Нет лишних полей за пределами схемы

---

#### TC-CLUB-C02: Успешное создание — content

**Проверки:**
- `bookTitle` == `"Война и мир"`
- `bookAuthors` == `"Лев Толстой"`
- `publicationYear` == `1869`
- `description` == `"Классика русской литературы"`
- `owner` == ID текущего пользователя
- `reviews` == `[]`

---

#### TC-CLUB-C03: Создание без авторизации

**Предусловия:** запрос без заголовка `Authorization`.

**Проверки:**
- Status code: `401`

---

#### TC-CLUB-C04: Создание без обязательного поля `bookTitle`

**Тело запроса:** все поля кроме `bookTitle`.

**Проверки:**
- Status code: `400`
- Ошибка валидации для `bookTitle`

---

#### TC-CLUB-C05: Невалидный `telegramChatLink` (не URI)

**Тело запроса:** `telegramChatLink` = `"not-a-url"`, остальные валидны.

**Проверки:**
- Status code: `400`
- Ошибка валидации для `telegramChatLink`

---

#### TC-CLUB-C06: `bookTitle` длиной 256 символов (> maxLength=255)

**Проверки:**
- Status code: `400`
- Ошибка валидации `bookTitle` по длине

---

#### TC-CLUB-C07: Пустое тело запроса

**Тело запроса:** `{}`

**Проверки:**
- Status code: `400`
- Ошибки для всех обязательных полей

---

### 3.2 Просмотр

#### TC-CLUB-R01: Список клубов — status + schema (`GET /api/v1/clubs/`)

**Предусловия:** существует хотя бы один клуб.

**Проверки:**
- Status code: `200`
- Тело содержит: `count` (int), `next` (string/null), `previous` (string/null), `results` (array)
- Каждый элемент `results` содержит все обязательные поля Club
- Нет лишних полей на уровне пагинации и на уровне объекта Club

---

#### TC-CLUB-R02: Список клубов — content

**Проверки:**
- `count` >= 1
- `results[0]` содержит данные ранее созданного клуба (bookTitle, bookAuthors — не stub/заглушки)

---

#### TC-CLUB-R03: Клуб по ID — status + schema (`GET /api/v1/clubs/{id}/`)

**Предусловия:** клуб с известным ID существует.

**Проверки:**
- Status code: `200`
- Тело содержит все обязательные поля Club
- Нет лишних полей

---

#### TC-CLUB-R04: Клуб по ID — content

**Проверки:**
- `id` == запрошенный ID
- `bookTitle`, `bookAuthors`, `description` — соответствуют данным при создании

---

#### TC-CLUB-R05: Клуб по несуществующему ID

**Входные данные:** `GET /api/v1/clubs/999999/`

**Проверки:**
- Status code: `404`

---

#### TC-CLUB-R06: Пагинация

**Предусловия:** в системе > 1 клуба.

**Входные данные:** `GET /api/v1/clubs/?page=1&page_size=1`

**Проверки:**
- Status code: `200`
- `results` — массив длиной 1
- `next` — не null (есть следующая страница)
- `count` > 1

---

### 3.3 Редактирование

**Предусловия:** пользователь авторизован и является owner клуба.

**Тело запроса:** `PUT /api/v1/clubs/{id}/`

```json
  "bookTitle": "Анна Каренина",
  "bookAuthors": "Лев Толстой",
  "publicationYear": 1878,
  "description": "Обновлённое описание",
  "telegramChatLink": "https://t.me/anna_k"
}
```

**Проверки:**
- Status code: `200`
- Тело содержит все обязательные поля Club
- Нет лишних полей

---

#### TC-CLUB-U02: PUT обновление — content

**Проверки:**
- `bookTitle` == `"Анна Каренина"`
- `publicationYear` == `1878`
- `description` == `"Обновлённое описание"`
- `modified` != null

---

#### TC-CLUB-U03: PATCH обновление — status + schema

**Тело запроса:** `PATCH /api/v1/clubs/{id}/`

**Проверки:**
- Status code: `200`

---

#### TC-CLUB-U04: PATCH обновление — content

**Проверки:**
- `description` == `"Только описание изменено"`

#### TC-CLUB-U05: Обновление чужого клуба

#### TC-CLUB-U06: Обновление без авторизации

#### TC-CLUB-U07: PUT с невалидными данными (пустой `bookTitle`)

**Тело запроса:** `bookTitle` = `""`, остальные валидны.

**Проверки:**
- Status code: `400`

## Техники тест-дизайна

- **Классы эквивалентности** — валидные / невалидные наборы данных, существующий / несуществующий пользователь.
- **Граничные значения** — username 150/151, bookTitle 255/256.
- **Разделение Schema / Content** — два отдельных ТК позволяют точно диагностировать: упал контракт или бизнес-логика.
- **Негативное тестирование** — отсутствие полей, невалидные данные, отсутствие авторизации, доступ к чужим ресурсам, несуществующие ID, повторные операции.
